# The tracker is the database: Effort, Spec, and Overrides live in the workspace's tracker

A workspace binds to exactly one tracker — GitHub Issues or Linear, behind a TrackerAdapter seam — and trackers are never mixed within an Effort (no native cross-tracker sub-issues or dependencies exist; owning the effort graph ourselves would reintroduce the drift this ADR forbids).

Threadmap derives all pipeline state from artifacts rather than keeping a state DB. Taken to its conclusion: an Effort *is* its map issue in the user-picked home repo (`home-repo#number` is its id); the Spec is a `threadmap:spec` sub-issue whose closing is the approval event — not a repo file, deliberately, so it self-archives as a closed, timestamped issue instead of rotting in `docs/` looking authoritative; Overrides are `threadmap:override:<stage>` labels on the map issue with structured audit comments; and member repos are derived from ticket ownership, never declared. Durable knowledge graduates to CONTEXT.md/ADRs — Specs are per-effort and disposable. The local `~/.threadmap` dir holds only ephemeral session data and a pointer cache, never authoritative state: two machines pointed at the same repos must always agree.
