# One derived Stage per Effort; Setup is Repo readiness, not a stage

Each Effort has exactly one current Stage (planning → to-spec → to-tickets → implement → code-review), computed purely from artifacts — no event ever sets it directly, so it can never drift. We rejected per-repo stage machines: multi-repo progress differences (one repo's PR in review while another is still implementing) are detail *inside* a stage, aggregated by effort-wide gates such as "every ticket has a linked PR" (implement) and "every PR merged with zero unresolved threads" (code-review). Setup (skills installed, per-repo docs generated) is a property of a Repo that gates Effort creation, not the pipeline's first stage — it belongs to repos, not features. Planning children (`wayfinder:*`) and implementation tickets (`threadmap:ticket`) are distinguished by label namespace so gates can count them separately off one issues query. To-tickets is the one gate with an explicit human sign-off (`threadmap:ticketed` stamp): spec coverage isn't machine-checkable, and it is the last checkpoint before agents write code.
